## 一、项目概述（面试时 30-60 秒说明）

- 项目名：hm-dianping（仿大众点评的练手项目）。
- 技术栈：Spring Boot、MyBatis-Plus、Redis（StringRedisTemplate/Redisson）、MySQL、Lombok、Hutool、Maven。
- 主要功能：用户登录、店铺管理、关注（Follow）、秒杀/抢购券（seckill）与下单（voucher order）、文件上传、分页/滑动查询等。
- 亮点：使用 Redis 做关注集合、秒杀使用 Lua 脚本与 Redis 做流控、使用 Redisson 做分布式锁、将热点数据缓存和数据库访问分离。

## 三、面试官可能问的常规问题（与简答要点）

1. 你在这个项目中负责什么模块？
   - 简述你实际动手的功能（例如实现关注、Redis 缓存、voucher 下单、集成 Redisson、写 Lua 脚本、接口设计与测试等）。

2. 项目用到了哪些中间件？为什么选这些？
   - Redis：缓存、计数、集合操作（求交集）、秒杀限流与消息队列（可拓展）。
   - Redisson：分布式锁，解决并发下的数据一致性和幂等问题。
   - MySQL：关系数据持久化，用 MyBatis-Plus 简化 CRUD。

3. 介绍 `FollowServiceImpl` 的实现与设计思路。
   - 关注/取关同时更新数据库与 Redis 集合（sadd/srem）以支持交集计算。
   - `isFollow` 直接查库判断；`followCommons` 利用 Redis 的 set 交集（SINTER）提高性能。

4. 项目如何处理高并发（例如秒杀）？
   - 使用 Redis + Lua 脚本实现原子检查与扣库存；把下单请求异步化（消息队列或 Redis Stream）；使用分布式锁/信号量防止超卖；采用预减库存/库存分段等策略。

5. 描述一次事务边界与你如何保证一致性。
   - 尽量将需要原子操作的操作包在数据库事务中；对于跨系统（DB + Redis）的操作，采用先 DB 后缓存或先缓存后 DB 的策略并用补偿/消息最终一致性机制。

6. 系统如何保证缓存与数据库的一致性？
   - 常见策略：缓存失效（先更新 DB，后删除缓存），双写（不推荐），使用消息队列异步更新缓存，或使用互斥锁+重建机制。项目中对关键读场景使用缓存预热与失效策略。

7. 你如何防止 Redis 的缓存穿透/击穿/雪崩？
   - 穿透：使用布隆过滤器或对空结果缓存（短时）
   - 击穿：使用互斥锁或单人重建缓存
   - 雪崩：缓存加随机过期、分散重建时间、使用主从、故障转移

8. 说说 MyBatis-Plus 带来的好处与局限。
   - 好处：减少 boilerplate、内置条件构造器（QueryWrapper）、分页方便
   - 局限：复杂 SQL 仍需 XML 手写、对复杂关联查询不好优化

## 四、场景题 / 八股文（带回答要点）

场景题 1 — 关注并发场景：同时大量用户关注/取消同一个人，可能出现异常或数据不一致，你如何处理？
- 要点：使用 DB 约束（唯一索引）避免重复插入；关注操作先写 DB（事务），再更新 Redis；在高并发下可使用异步队列缓冲 DB 写入或使用乐观锁/去重；对 Redis 操作使用幂等写入（sadd 本身幂等）；对删除使用 srem。

场景题 2 — `followCommons` 返回是空或少于预期（交集操作结果与 DB 不一致），可能的原因与排查步骤？
- 要点：可能是 Redis 未及时更新（关注/取消未写入或失败），也可能是用户数据被删除/ID 类型转换异常。排查：检查 Redis key 是否存在并查看集合内容，检查相关关注操作的异常日志，确认 Redis 与 DB 的最终一致性策略。

场景题 3 — 秒杀场景出现超卖，你如何定位并解决？
- 要点：超卖可能来自并发读写竞态、事务隔离不当或缓存预减失败。定位：复现压测、查看 Lua 脚本逻辑、检查是否有多个并行 DB 写路径、检查库存来源。解决：使用 Redis 原子脚本做库存扣减+下单消息写入队列；后端异步消费并加分布式锁保证一个订单只会被消费一次；DB 侧使用乐观锁（version）/库存校验。

场景题 4 — 如果 Redis 不可用，系统如何降级？
- 要点：采用降级策略：读直接回源（DB），写入队列或直接失败并给用户友好提示；熔断/限流降低系统压力；关键业务开启多级缓存或本地缓存作为缓冲。

场景题 5 — 解释一下你在项目中使用的 Lua 脚本（`seckill.lua`）做了什么，为什么放在 resources？
- 要点：Lua 脚本在 Redis 中原子执行，常用于原子检查库存、扣减并写入流水、限制每用户秒杀次数等。放 resources 便于维护、热加载与和 Java 端读取注册脚本。

## 五、代码级问题（面试官可能会要求你解释或修改）

1. 请解释 `FollowServiceImpl.follow` 的实现，并指出潜在问题或改进点。
   - 说明：当前实现同时写 DB（save/remove）与 Redis（sadd/srem）。
   - 改进：对 save 操作加入唯一索引异常处理以防重复插入；判断 `UserHolder.getUser()` 为空的容错；考虑写失败回滚 Redis 或 retry 机制；在高并发下可将 DB 写入异步化到 MQ 来削峰。

2. `isFollow` 方法中使用 `query().count()`，效率如何？能否改进？
   - 回答要点：count 会走数据库聚合，若只需判断是否存在，可使用 `exists` 或 `select 1 limit 1` 的方式减少开销；MyBatis-Plus 可用 `getOne` 并限制只返回一条。

3. `followCommons` 返回的 `userService.listByIds(ids)` 顺序和 `ids` 是否对应？如何保证顺序？
   - 回答要点：`listByIds` 通常不保证顺序；如果面试官关心顺序，可在查询后按 `ids` 顺序做二次排序或使用 `ORDER FIELD`（MySQL）查询确保顺序。

4. 如果 Redis 集合非常大（百万级），`SINTER` 性能问题怎么办？
   - 回答要点：避免一次性 SINTER 大集合，可使用预计算共同好友、限制集合大小、分页求交或采用位图/布隆/倒排索引策略，或把热度用户用特殊缓存结构处理。
