# LSMarket 两周升级开发计划（Kafka 版）

## 1. 文档目标

本计划用于指导 `LSMarket` 在 14 天内完成高并发能力升级，重点覆盖以下目标：

- 将秒杀异步链路从 Redis Stream 升级为 Kafka（保留可回滚开关）
- 补齐消息可靠性（重试、死信、幂等）
- 增强高并发保护（限流、降级）
- 建立可观测性与压测基线，形成可量化结果
- 产出可直接用于面试表达的项目材料

---

## 2. 当前现状与问题

基于现有代码（`VoucherOrderServiceImpl` + `seckill.lua`）可确认：

- 秒杀资格校验和扣库存已在 Lua 中原子执行
- 异步下单依赖 Redis Stream 消费
- 当前链路存在可靠性风险和工程化不足

需要优先处理的问题：

1. `XACK` 的 stream 名与读取的 stream 名不一致（`stream.orders` vs `s1`），会导致消息确认异常。
2. 缺少明确的消息重试上限与死信处理机制。
3. 消费幂等没有数据库唯一索引兜底（建议 `(user_id, voucher_id)`）。
4. 压测和监控链路不完整，优化效果难量化。

---

## 3. 技术方案总览

### 3.1 MQ 选型结论

本次升级采用 **Kafka**，理由：

- 与现有项目方向（高并发、吞吐优先）匹配
- 面试表达更统一（可与 `PaiSmart` 的 Kafka 经验联动）
- 可自然扩展到重试 Topic、死信 Topic、消费组扩展

### 3.2 核心设计原则

- 先保证正确性，再追求吞吐
- 至少一次投递 + 消费幂等（不强追 exactly-once）
- 全链路可观测（指标可看、故障可追、问题可复盘）
- 全程保留回滚能力（`mq.type` 配置开关）

### 3.3 升级后架构（简版）

1. 请求进入秒杀接口  
2. Lua 原子校验库存与一人一单  
3. 校验成功后发布 Kafka 订单事件  
4. Kafka 消费端异步创建订单  
5. 失败消息进入重试 Topic，超过阈值进入死信 Topic  
6. 消费端依赖幂等策略确保不重复下单

## 4. 14 天排期（按天执行）

## 第 1 周：完成迁移主链路和可靠性骨架

### Day 1：基线建立 + 当前链路修复

- 修复 Redis Stream `XACK` 的 stream 名不一致问题
- 固化压测脚本（秒杀接口）
- 记录优化前基线：QPS、P95、错误率、成功下单率
- 产出《基线压测报告 v1》

交付物：

- 修复提交
- 压测脚本（可重复执行）
- 基线数据表格

### Day 2：幂等基础建设

- 数据库增加唯一索引：`tb_voucher_order(user_id, voucher_id)`
- 消费下单逻辑补幂等兜底（捕获重复键异常）
- 明确业务幂等日志字段（订单ID、用户ID、券ID）

交付物：

- SQL 变更脚本
- 幂等处理代码

### Day 3：消息层抽象

- 抽象消息发布与消费接口：
  - `OrderEventPublisher`
  - `OrderEventConsumer`
- 先保留 Redis Stream 实现，再新增 Kafka 实现
- 配置增加 `mq.type=redis|kafka`

交付物：

- 抽象接口 + 两套实现骨架
- 配置开关

### Day 4：Kafka 生产端接入

- 新增订单事件模型 `OrderCreatedEvent`
- 秒杀成功后发布到 Kafka Topic：`seckill-order-created`
- 失败发布告警与监控计数

交付物：

- Kafka Producer
- 事件模型与序列化配置

### Day 5：Kafka 消费端接入

- 新增 `@KafkaListener` 消费逻辑
- 处理成功后提交 offset（手动提交）
- 下单异常进入重试流程

交付物：

- Kafka Consumer
- 消费与业务处理解耦实现

### Day 6：重试与死信

- 新增重试 Topic：`seckill-order-retry`
- 新增死信 Topic：`seckill-order-dlt`
- 实现最大重试次数（建议 3 次）与指数退避
- 提供死信重放脚本或管理接口

交付物：

- Retry + DLT 机制
- 死信处理说明文档

### Day 7：阶段验收

- 执行第 1 次迁移后压测（同 Day 1 脚本）
- 对比 Redis Stream 与 Kafka 版本表现
- 产出《阶段验收报告 v1》

交付物：

- 压测对比表
- 问题清单与下周修复计划

## 6. 验收标准（Definition of Done）

满足以下条件视为升级完成：

1. 功能正确性
- 不超卖
- 不重复下单
- 秒杀高峰期系统可降级而非崩溃

2. 可靠性
- 消费失败可重试
- 超限消息进入 DLT 且可重放
- 消费幂等可证明

3. 性能与稳定性
- 有优化前后压测对比
- 核心指标可被监控系统观测

4. 可讲述性
- 能用 2 分钟讲清升级背景、方案、结果
- 追问可覆盖限流、幂等、重试、一致性

---

## 7. 风险与应对

- 风险：Kafka 接入影响现有链路稳定性  
应对：保留 `mq.type` 开关，支持快速切回 Redis Stream。

- 风险：迁移期间重复下单  
应对：数据库唯一索引强约束 + 消费端幂等处理。

- 风险：重试风暴导致雪崩  
应对：指数退避 + 最大重试次数 + DLT 截断。

- 风险：优化无量化证明  
应对：固定压测脚本与基线，所有优化必须以数据对比收口。
